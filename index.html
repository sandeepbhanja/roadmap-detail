W9
Vector Similarity — RAG Foundation
RAGEMBEDDINGSPGVECTOR
▾
WHY THIS WEEK EXISTS
This is where GenAI enters the system. The insight: you've seen thousands of incidents before. When a new incident arrives, the fastest way to understand it is to find the most similar past incident and look at how that was resolved. This is RAG — Retrieval Augmented Generation. You retrieve relevant context from your own historical data, then use it to generate better answers. Every enterprise AI product uses this pattern.
✨ RAG + EMBEDDINGS — pgvector Similarity Search
How it works: Each incident is embedded using OpenAI text-embedding-3-small (or a local model). The embedding is a 1536-dimensional vector that captures semantic meaning. When a new incident arrives, you embed it and run: SELECT * FROM incident_embeddings ORDER BY embedding <=> $1 LIMIT 5. The <=> operator is cosine distance — finds the 5 most semantically similar past incidents. Their resolved root causes become evidence for ranking the current incident's root cause. This is RAG applied to operations data.

W12
[AGENT] 3-Layer Memory Architecture
AGENTMEMORYREDIS + PGVECTOR
▾
WHY THIS WEEK EXISTS
LLMs have no memory between calls. You build it. Three layers because different information has different access patterns: Redis for the current investigation (sub-millisecond, gone after 4h), pgvector for past incidents (semantic search, permanent), PostgreSQL graph for service topology (structured, queryable). This is exactly how production agent systems like Claude's memory or Cognition's Devin work.
✨ MEMORY ARCHITECTURE — 3 Layers
Layer 1 Redis (Short-term): Current incident trace — every T/A/O step stored as list, TTL=4h. Layer 2 pgvector (Long-term): All resolved incidents as embeddings — searchable by semantic similarity. This is where past postmortems live. Layer 3 PostgreSQL Graph (Structured): Service dependency topology — the "world model" of your system. Different agents read different layers — enforced by MemoryAccessPolicy.

W14
[AGENT] Tree-of-Thoughts Branching
AGENTTREE-OF-THOUGHTS
▾
WHY THIS WEEK EXISTS
Linear reasoning gets stuck. If the first hypothesis is wrong, a ReAct agent can waste all its steps going down that path. Tree-of-Thoughts solves this: the Hypothesis Agent generates 3 competing theories simultaneously. The Verification Agent tests all 3 in parallel. The best-scoring branch wins. It's like having 3 engineers independently investigating and then comparing notes — much more robust than one engineer making sequential guesses.
🌿 TREE-OF-THOUGHTS — Parallel Hypothesis Evaluation
Hypothesis Agent prompt: "Generate 3 distinct, mutually exclusive hypotheses for this incident. Return JSON array." Each branch gets its own tool calls. Score = evidence_matches(50%) + prior_confidence(30%) + similar_past_incidents(20%). Parallel execution with CompletableFuture.allOf(). If top-2 scores within 0.1 of each other → "ambiguous" flag → human review. This is research-grade reasoning, productionized.
W15
[AGENT] Reflection Agent (Reflexion)
AGENTSELF-IMPROVEMENTLONG-TERM MEMORY
▾
WHY THIS WEEK EXISTS
Most AI systems are static — they don't get better from their mistakes. Reflexion changes this. After each incident closes, the Reflection Agent reads the full reasoning trace, compares predicted vs actual root cause, generates a verbal "lesson learned," stores it in long-term memory, and adjusts the scoring weights. Future incidents will use these lessons. The system literally gets smarter from its own errors over time.
🪞 REFLEXION — Verbal Self-Improvement Loop
Reflection Agent prompt: "Here is the reasoning trace for incident X. The predicted root cause was payment-service (78%), but the actual root cause was db-connection-pool. What went wrong in the reasoning? What should the agent do differently next time?" The generated reflection note is embedded and stored in pgvector — future similar incidents retrieve these lessons as additional context. Weights auto-adjust based on rolling accuracy. This is the Reflexion paper in production code.

W22
LSTM Forecasting
DEEP LEARNINGPYTORCHFASTAPI
▾
WHY THIS WEEK EXISTS
AR models assume linear relationships — demand next week is a linear combination of past demand. LSTM learns non-linear patterns: seasonal spikes, weather dependencies, holiday effects. The feature vector (14 inputs including lag features and rolling statistics) is your first real "feature engineering" for deep learning. The Monte Carlo dropout gives you uncertainty estimates — critical for the RL agent to know when to explore more.
🧠 DEEP LEARNING MODEL — LSTM with Uncertainty
Architecture: LSTM(64) → Dropout(0.2) → LSTM(32) → Dense(1). Input: 14-day sequence of [demand, price, weather, day_of_week, rolling_mean_7, rolling_std_7]. Output: demand for day 15 + confidence interval via Monte Carlo dropout (50 forward passes). Deployed as FastAPI service. Integrated into the model auto-selector alongside MA and AR — whichever has lowest recent MAE gets used.
W23–25
RL Foundations → Tabular Q → Deep Q-Network
REINFORCEMENT LEARNINGDQNPYTORCH
▾
WHY THIS WEEK EXISTS
Forecasting tells you what demand will be. But what price should you set? This is a sequential decision problem — today's price affects tomorrow's demand. RL is the right framework: an agent (your pricing policy) takes actions (set price), receives rewards (revenue), and learns which actions are best in which situations. You build tabular Q-learning first (to understand the algorithm) then DQN (to handle continuous state spaces). This is exactly how Uber's surge pricing and airline yield management systems work.
🎮 REINFORCEMENT LEARNING — DQN Pricing Agent
MDP Formulation: State = {demand_forecast, current_price, competitor_price, weather, hour_of_day}. Action = price multiplier {0.8, 0.9, 1.0, 1.1, 1.2}. Reward = revenue - competitor_penalty. DQN: Neural net (128→64→5) replaces Q-table for continuous states. Experience replay buffer (10k transitions) breaks correlation. Target network updated every 100 steps for stability. After 2000 training episodes, the policy learns: raise price when demand is high AND competitor price is high, lower when weather is bad. The agent discovers elasticity — without being told.

[AGENT] Forecasting Agent — Dynamic Model Selection
AGENTLSTM + AR + MA
▾
WHY THIS WEEK EXISTS
A static model selector (always use LSTM) is brittle. After a demand drift event, AR is actually more robust. The Forecasting Agent wraps all three models as tools, detects drift in real-time, and picks the best model for the current distribution. W32
[AGENT] RL + LLM Hybrid Pricing Agent
RLLLM RISK EVALHYBRID POLICY
▾
WHY THIS WEEK EXISTS
Pure RL is blind to text. It doesn't know a competitor just went bankrupt (opportunity!) or a major storm is coming (reduce prices!). LLMs can read this context but can't learn a pricing policy. The hybrid combines both: RL proposes a base price from its learned policy, the LLM reads the market context and outputs a risk multiplier (0.7–1.3×), the final price is their product. This is the frontier of production AI systems — combining the learning of RL with the world-knowledge of LLMs.
💬 LLM USAGE — Risk Multiplier for RL Policy
LLM prompt: "You are a pricing risk evaluator. Given: demand_forecast=HIGH, weather=STORM, competitor_news='main competitor shut down operations'. Output JSON: {risk_multiplier: float(0.7-1.3), reasoning: str}." The LLM says: "Competitor shutdown is a demand surge opportunity. Despite storm, raise prices. Multiplier: 1.25." Final price = DQN_base_price × 1.25. Cached for 15min in Redis. Guardrails: max 20% change per step, floor/ceiling enforced. LLM caching adds the text-awareness RL doesn't have.
W33
[AGENT] Market Event Agent — News Embeddings
EMBEDDINGSAGENTCLUSTERING
▾
WHY THIS WEEK EXISTS
External signals — a news headline about a competitor, a government policy change, a weather forecast — directly affect optimal pricing. This agent processes these unstructured text signals using embeddings and clustering, then converts them into structured causal tags (demand-up / demand-down / neutral) that the RL agent can use. ✨ EMBEDDINGS — News Event Clustering

This is RAG applied to market intelligence — unstructured text becomes structured signals for your ML model.
W34
[AGENT] Strategy Reflection Agent
AGENTSELF-IMPROVEMENTREWARD SHAPING
▾
WHY THIS WEEK EXISTS
This is the most advanced concept in your entire roadmap: an agent that modifies the reward function of another agent. After each simulated day, the Reflection Agent looks at profit vs forecast accuracy and adjusts the RL agent's reward weights — more competitor penalty if you're losing to competitors, less price weight if forecast is bad. The RL agent then retrains on the adjusted reward. This is meta-learning: a system that learns how to learn. Almost no portfolio projects have this.
🔄 META-LEARNING — Self-Adjusting Reward Function
Reflection Agent reads: daily_revenue, forecast_MAPE, competitor_loss_events. Rules: if profit_delta < -5% → increase competitor_penalty by 0.1. If MAPE > 15% → increase demand_weight, reduce price_weight. All weights clamped [0.1, 2.0]. New weights versioned in reward_params table. Next RL training episode initializes with updated reward function. Plot over 30 days: reward weights + accuracy converge. This is "learning to optimize the optimizer."

Why you use 6 different AI techniques (not just one LLM)
Each technique has a different strength. LLMs are good at language and reasoning but can't learn from data. RL is good at sequential decisions but can't read text. ML models (LSTM, IF) learn patterns from data but can't explain themselves. Embeddings capture semantic meaning. RAG retrieves relevant context. Agents coordinate all of these. A senior AI engineer knows which tool to use for which problem — and that's exactly what you're learning.

Hybrid Policy Agent
The Pricing Agent combines RL (data-driven sequential policy) with LLM (text-driven risk assessment). RL proposes the base price from its trained policy. LLM reads market context and outputs a risk multiplier. Final price = RL × LLM. Each component does what it's good at — RL learns from data, LLM understands language.

P2 · W34 — Meta-Learning
Agent That Modifies Another Agent's Reward
The Reflection Agent modifies the reward function weights that the RL Pricing Agent trains on. This is meta-learning: the system learns how to shape the learning process of its own sub-agent. Each training episode uses the latest reward function version. This is the most sophisticated AI pattern in your entire portfolio.

P1 · W5 — EWMA + Z-Score
Statistical Anomaly Detection
Exponentially weighted moving average for baseline, Welford online algorithm for standard deviation, Z-score threshold (3σ) for anomaly detection. Fast (microseconds), no model loading, interpretable. First line of defence — catches sudden spikes that the ML model might miss on sparse data.

Multivariate Anomaly Detection
Detects anomalies that are multi-dimensional (abnormal error_rate AND abnormal latency simultaneously). Trained on 7-day sliding window. Deployed as FastAPI microservice. Combined with Z-score via strategy pattern — either detector can flag. Ensemble approach reduces both false positives and false negatives.

P2 · W21 — MA + AR Forecasting
Statistical Demand Forecasting
Moving average (simple baseline) and autoregressive AR(7) model. Trained with OLS. These are the benchmarks your LSTM must beat. Also serve as fallback when LSTM fails or when drift is detected (AR is more robust under distribution shift). Auto-model selector picks the lowest 14-day MAE.
statsmodelsOLSMAE auto-selection
P2 · W22 — LSTM
Deep Learning Demand Forecast
LSTM(64) → Dropout → LSTM(32) → Dense(1). Learns non-linear temporal patterns including weather-demand coupling, day-of-week seasonality, lag dependencies. Monte Carlo dropout (50 forward passes) gives uncertainty estimates. Deployed as FastAPI service integrated with the Forecasting Agent's auto-selector.

P2 · W24 — Tabular Q-Learning
RL: Learn Pricing Policy (Simple)
Q-table (state_space × action_space). Q-update: Q(s,a) += α[r + γ×max(Q(s',·)) - Q(s,a)]. Epsilon-greedy exploration decaying from 1.0 to 0.05. 2000 training episodes. Teaches you exactly how Q-learning works before DQN abstracts it away with a neural network.
numpy Q-tableepsilon-greedyPostgreSQL persistence
P2 · W25 — Deep Q-Network
RL: Neural Network Pricing Policy
Neural net replaces Q-table for continuous state spaces. Experience replay buffer (10k transitions) breaks sequential correlation. Target network (updated every 100 steps) stabilizes training. Learns the full pricing policy: when to raise, hold, or discount based on demand, competition, and weather — without being programmed with rules.
